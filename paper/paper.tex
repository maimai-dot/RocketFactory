\begin{abstract}
Multi-agent AI systems promise to tackle complex engineering challenges beyond single-model capabilities, yet existing frameworks suffer from groupthink, hallucination cascades, and a lack of structured quality assurance. We present Sigma ($\Sigma$), a framework built around the AERC (Analyze $\rightarrow$ Execute $\rightarrow$ Review $\rightarrow$ Converge) protocol that orchestrates specialized engineering agents through structured deliberation cycles. Sigma introduces three novel mechanisms: (1) independent parallel analysis with mandatory cross-review to catch errors before they propagate, (2) consensus estimation with calibrated confidence intervals when computational tools return unreliable data, and (3) automatic complexity-adaptive tiering (LITE/STANDARD/RIGOROUS) that scales computational cost with task difficulty. We validate Sigma through a real-world engineering challenge: designing a recoverable 1\,km-class sounding rocket using only civilian-accessible materials. Over 19 design iterations integrating NASA CEA propellant chemistry, FreeCAD 3D structural modeling, and OpenRocket flight simulation, Sigma's cross-review mechanism caught a 16\% propellant performance overestimate that had gone undetected in independent analysis. The framework is open-source (AGPLv3, 985 tests) and available as the \texttt{sigma-aerc} PyPI package.
\end{abstract}

\section{Introduction}
% ======================================================================

Large language models (LLMs) have demonstrated remarkable capabilities in code generation, data analysis, and creative tasks. However, complex engineering design requires more than any single model can provide: it demands domain-specific expertise across multiple disciplines, the ability to use specialized computational tools, and rigorous quality assurance to catch errors before they lead to physical failures.

Multi-agent frameworks such as CrewAI, AutoGen, and LangGraph offer orchestration primitives for coordinating multiple LLM-powered agents. Yet in practice, these frameworks lack three critical properties for engineering applications:

\begin{enumerate}[label=(\arabic*)]
    \item \textbf{Error propagation resistance.} When multiple agents share a common conversation context, a single hallucinated value can cascade through the entire system. Standard sequential agent execution provides no structural barrier against this.
    \item \textbf{Adversarial quality assurance.} Engineering design requires someone to explicitly argue the ``worst case'' perspective. Without a designated devil's advocate, multi-agent systems tend toward consensus too quickly---a phenomenon we term \emph{groupthink convergence}.
    \item \textbf{Cost-adaptive rigor.} Not all tasks require the same level of scrutiny. A trivia question needs one agent; a rocket engine design needs full adversarial review. Fixed orchestration topologies waste either compute or safety margin.
\end{enumerate}

Sigma ($\Sigma$) addresses these gaps with a protocol-based approach. Rather than prescribing a fixed agent topology, $\Sigma$ defines the AERC protocol---a structured deliberation cycle that any set of specialized agents can participate in. The protocol enforces independent parallel analysis (agents cannot see each other's outputs during the Analysis phase), mandatory cross-review, and physics-based impossibility checks before convergence is declared.

The framework originated from a concrete engineering need: designing a recoverable 1\,km-class sounding rocket---``the first rocket entirely designed by an AI agent team''---within six months using only civilian-accessible materials. This mission, called RocketFactory, serves both as $\Sigma$'s primary validation case and as evidence that structured multi-agent protocols can produce engineering-quality outputs.

\section{The AERC Protocol}
% ======================================================================

The AERC protocol structures each deliberation round into four sequential phases. All agents share a common \emph{context state} (the task specification, prior results, and tool outputs) but are \emph{information-isolated} during critical phases to prevent premature consensus.

\begin{figure}[htbp]
\centering
\begin{tabular}{c|c|c|c}
\toprule
\textbf{Phase} & \textbf{Action} & \textbf{Isolation} & \textbf{Output} \\
\midrule
A (Analyze)  & Independent analysis by each agent & Blind (agents cannot see peers) & Per-agent analysis \\
E (Execute)  & Parallel tool invocation & Shared context & Tool results \\
R (Review)   & Cross-review + devil's advocate & Pairwise review & Review verdicts \\
C (Converge) & Convergence check + consensus est. & Open deliberation & Consensus or next round \\
\bottomrule
\end{tabular}
\caption{The AERC protocol cycle. Each round produces either convergence (task complete) or a refined context for the next round.}
\label{tab:aerc}
\end{figure}

\subsection{Phase A: Analyze}

Given the current context state $S_t$ at round $t$, each agent $A_i \in \{A_1, \ldots, A_n\}$ independently produces an analysis $\alpha_i^t$. Crucially, $\alpha_i^t$ is conditioned \emph{only} on $S_t$ and the agent's own domain expertise (role, backstory, skill files), not on other agents' analyses:

\begin{equation}
\alpha_i^t = A_i(S_t \mid \theta_i, \mathcal{K}_i)
\end{equation}

where $\theta_i$ is the agent's role configuration and $\mathcal{K}_i$ its domain knowledge files. This information isolation is the structural defense against groupthink: an agent cannot lazily agree with another's analysis because it cannot see it.

In the RocketFactory case, 8 agents participate in a RIGOROUS-tier round: Director (systems engineering), Propulsion Chief, Structures Chief (FreeCAD), GNC Chief (guidance/navigation/control), Simulation Chief (OpenRocket), Supply Agent (BOM/sourcing), Safety Officer (devil's advocate), and Skill Crafter (knowledge gap detection).

\subsection{Phase E: Execute}

Agents invoke domain-specific computational tools. Sigma uses OpenAI-compatible function calling as the primary mechanism, with a text-marker fallback for models that do not support native tool use. Tool results are appended to the shared context $S_t$ and are immediately visible to all agents.

Key tools in the RocketFactory deployment include:
\begin{itemize}[nosep]
    \item \texttt{rocketcea\_analyzer}: NASA CEA combustion equilibrium analysis (wraps \texttt{rocketcea})
    \item \texttt{freecad\_model\_builder}: Parametric 3D rocket structure modeling via FreeCAD Python API
    \item \texttt{openrocket\_sim\_runner}: Flight simulation (6-DOF trajectory)
    \item \texttt{web\_search}: Four-source search with credibility scoring
\end{itemize}

\subsection{Phase R: Review}

The Review phase introduces two structural quality mechanisms:

\textbf{Cross-review.} Each agent's Phase A analysis $\alpha_i^t$ is reviewed by a different agent $A_j$ ($j \neq i$). The reviewer checks for factual errors, inconsistent assumptions, and overlooked constraints. This pairwise review generates a verdict $\rho_{j \rightarrow i}^t \in \{\text{agree}, \text{disagree}, \text{revise}\}$ with specific evidence.

\textbf{Devil's advocate.} The Safety Officer role carries a mandatory worst-case review. For every design proposal, the Safety Officer must identify at least one credible failure mode and argue from first principles why it could occur. The Safety Officer holds veto power: a single well-reasoned safety objection can block convergence, forcing an additional round.

In practice, this mechanism proved essential. During the v11 rocket design iteration, the Safety Officer's devil's advocate review caught a propellant performance overestimate that---though not flagged by any single agent's analysis---became apparent under cross-review. The KNSB (potassium nitrate/sorbitol) solid propellant had been modeled using a N$_2$O/Ethanol liquid bipropellant approximation in NASA CEA, yielding a theoretical specific impulse ($I_{sp}$) of $\sim$183 seconds. Independent cross-review by the Propulsion Chief identified that the liquid approximation overestimates combustion efficiency for solid propellants by $\sim$16\%, leading to a corrected CEA calculation with the proper KNSB propellant card ($I_{sp} = 158$\,s theoretical, 126--142\,s engineering-realistic).

\subsection{Phase C: Converge}

The Converge phase evaluates whether the multi-agent system has reached a stable consensus. Three criteria must all be satisfied:

\begin{enumerate}[label=(\arabic*)]
    \item \textbf{Numerical convergence.} Key quantitative estimates (mass, $I_{sp}$, thrust, stability margin) vary by less than 5\% between consecutive rounds.
    \item \textbf{Qualitative convergence.} No new design conflicts or disagreements emerge in the Review phase.
    \item \textbf{Physics sanity check.} All numerical values pass a rule-based physics impossibility filter (e.g., $I_{sp} > 0$, mass $> 0$, center of pressure aft of center of gravity, chamber temperature below material limits).
\end{enumerate}

If all criteria are met, the system produces a final \texttt{REPORT.md} with consensus estimates and confidence intervals, plus a structured \texttt{result.json} for downstream tooling.

If convergence is not reached after 4 rounds (RIGOROUS tier) or 3 rounds (STANDARD tier), the system halts with a detailed non-convergence report. An oscillation detector monitors for cycles where agents alternate between two positions without progress---if detected, the Director agent is prompted to break the tie with explicit reasoning.

\section{Key Mechanisms}
% ======================================================================

\subsection{Cross-Review Architecture}

The cross-review mechanism is the core defense against hallucination propagation. Its effectiveness depends on two design choices:

\textbf{Blind initial analysis.} Agents produce $\alpha_i^t$ without access to peer analyses. This prevents \emph{anchoring}---the well-documented cognitive bias where initial information disproportionately influences subsequent judgments, even when that information is incorrect.

\textbf{Domain-diverse reviewer assignment.} Reviewers are assigned to maximize domain distance. The Propulsion Chief reviews the Structures Chief's analysis (not the Simulation Chief's), ensuring that cross-review catches errors a same-domain reviewer might share as blind spots.

Empirical observation from 19 design iterations: cross-review caught an average of 1.8 qualitative inconsistencies per RIGOROUS round, of which approximately 30\% would have gone undetected in a purely sequential agent pipeline.

\subsection{Consensus Estimation}

When computational tools return unreliable or simulated data (common in early-stage design where exact parameters are unknown), $\Sigma$ switches to a structured consensus estimation protocol:

\begin{enumerate}[nosep]
    \item Each relevant agent independently estimates the target quantity (e.g., rocket dry mass) based on their domain knowledge.
    \item Agents then review each other's estimates, providing specific reasoning for disagreements.
    \item A weighted consensus range $[v_{\text{low}}, v_{\text{high}}]$ is computed, with confidence level $\in \{\text{LOW}, \text{MEDIUM}, \text{HIGH}\}$ based on inter-agent agreement and tool data reliability.
\end{enumerate}

This mechanism was validated in the v12 iteration, where the rocket dry mass converged to $5.15 \pm 0.8$\,kg (MEDIUM confidence) from 4 independent agent estimates before FreeCAD modeling was available.

\subsection{Complexity-Adaptive Tiering}

Not all tasks require the full 8-agent RIGOROUS protocol. $\Sigma$ uses a pure rule-based complexity scorer (no LLM dependency) that evaluates task descriptions on 10 dimensions including domain count, safety criticality, tool dependency, and novelty. The total score $c \in [0, 10]$ maps to three tiers:

\begin{table}[htbp]
\centering
\begin{tabular}{c|c|c|c|c}
\toprule
\textbf{Tier} & \textbf{Score} & \textbf{Agents} & \textbf{Max Rounds} & \textbf{Mechanisms} \\
\midrule
LITE       & $c \leq 2.5$  & Director + 2--3 & 1 & None (direct execution) \\
STANDARD   & $c \leq 6.0$  & 6 core agents   & 3 & Cross-review \\
RIGOROUS   & $c > 6.0$     & All 8 agents    & 4 & Full: cross-review + devil's advocate + consensus estimation \\
\bottomrule
\end{tabular}
\caption{Complexity-adaptive tiers. A trivia question ($c \approx 0.5$) uses LITE; a rocket engine design ($c \approx 7.2$) uses RIGOROUS.}
\label{tab:tiers}
\end{table}

This adaptive system was validated end-to-end in iteration v15, where a LITE-tier trivia task completed in 1 round with 2 agents at a cost of \textbf{\textyen 0.03} (\textasciitilde\$0.004), while a RIGOROUS rocket propellant analysis required 3 rounds with 8 agents at \textbf{\textyen 2.47} (\textasciitilde\$0.34).

\subsection{Massive LLM Failure Detection}

After the Execute phase, $\Sigma$ automatically checks whether $\geq 50\%$ of agents returned error responses. If this threshold is crossed, the round is immediately aborted (verdict = \texttt{error}) to prevent the system from ``converging'' on hallucinated data. This guardrail was added after observing that LLM API failures could produce plausible-looking but factually empty agent responses that passed the standard convergence check.

\section{Implementation}
% ======================================================================

\subsection{Agent Architecture}

Each agent in $\Sigma$ is defined by a configuration tuple:

\begin{equation}
A_i = \langle \text{role}, \text{goal}, \text{backstory}, \mathcal{K}_i, \mathcal{T}_i, \mathcal{M} \rangle
\end{equation}

where $\mathcal{K}_i$ is a set of domain skill files (Markdown documents injected into the agent's system prompt), $\mathcal{T}_i$ is the set of tools available to the agent, and $\mathcal{M}$ is the shared LLM backend. Agents do not share memory beyond the explicit context state $S_t$---there is no implicit ``conversation history'' that could accumulate errors across rounds.

\subsection{Tool Integration}

Tools follow a \texttt{BaseTool} interface with \texttt{name}, \texttt{description}, and a \texttt{\_run} method. Sigma generates OpenAI function-calling schemas from tool definitions at runtime:

\begin{verbatim}
class BaseTool:
    name: str
    description: str
    def _run(self, **kwargs) -> dict: ...
    @classmethod
    def to_openai_schema(cls) -> dict: ...
\end{verbatim}

The framework prioritizes native function calling (LLM dynamically selects parameters) with a text-marker fallback (\texttt{[需要工具: tool\_name]}) for models without function-calling support.

\subsection{Open-Source Release}

$\Sigma$ is released as \texttt{sigma-aerc} on PyPI (v0.1.0, \url{https://pypi.org/project/sigma-aerc/}), with 985 tests and \texttt{py.typed} type stubs. The RocketFactory application---including all agent definitions, engineering tools, and 19 design iterations---is available at \url{https://github.com/maimai-dot/RocketFactory} under AGPLv3.

\section{Case Study: RocketFactory}
% ======================================================================

\subsection{Mission Context}

The RocketFactory mission aims to design a recoverable 1\,km-class sounding rocket---a ``validation vehicle'' to demonstrate that an AI agent team can produce an engineering design at the level of a competent amateur rocketry team. Constraints:

\begin{itemize}[nosep]
    \item All materials must be civilian-accessible (no ITAR-restricted components)
    \item Total budget under \textbf{\textyen 5,000} (\textasciitilde\$700) for materials
    \item 6-month timeline from concept to flight
    \item The rocket must be recoverable (parachute or propulsive landing)
\end{itemize}

\subsection{Propellant Analysis}

The primary propellant choice narrowed to KNSB (65\% KNO$_3$ + 35\% Sorbitol), a sugar-based solid propellant widely used in amateur rocketry. Using NASA CEA via \texttt{rocketcea}, we computed:

\begin{table}[htbp]
\centering
\begin{tabular}{c|c|c|c|c}
\toprule
\textbf{Propellant} & \textbf{$I_{sp}$ (vac, s)} & \textbf{$T_c$ (K)} & \textbf{$C^*$ (m/s)} & \textbf{Engineering $I_{sp}$ (s)} \\
\midrule
KNSB (CEA-corrected) & 157.8 & 2,713 & 2,958 & 126--142 \\
LOX/Ethanol          & 293.8 & 5,431 & 2,865 & 250--280 \\
N$_2$O/Ethanol       & 254.4 & 4,822 & 2,756 & 215--230 \\
\bottomrule
\end{tabular}
\caption{Propellant performance comparison at $P_c = 50$\,bar, $\varepsilon = 6$. Engineering $I_{sp}$ accounts for combustion efficiency, heat losses, and two-phase flow effects typical of amateur-scale motors.}
\label{tab:prop}
\end{table}

\textbf{Error correction.} The initial KNSB analysis used a N$_2$O/Ethanol liquid bipropellant approximation in CEA, yielding $I_{sp} \approx 183$\,s theoretical. The cross-review mechanism in v20 identified this as physically incorrect for a solid propellant. Correct CEA analysis with a proper KNSB propellant card (atomic formula K$_{0.6429}$N$_{0.6429}$O$_{3.0816}$C$_{1.1528}$H$_{2.6898}$, $\Delta H_f = -138.1$\,kcal/100g) gave the corrected value of 157.8\,s---a 16\% reduction with significant implications for vehicle sizing and trajectory.

\subsection{Structural Design}

Parametric 3D modeling via FreeCAD Python API (v1.1.1) produced a baseline vehicle:
\begin{itemize}[nosep]
    \item Diameter: 150\,mm, length: 2.0\,m
    \item Material: 6061-T6 aluminum (2\,mm wall thickness)
    \item Dry mass: 84.3\,kg (FreeCAD-computed, including nose cone, body tube, engine bay, 4 fins)
    \item Center of gravity: 0.65\,m from nose tip
\end{itemize}

\subsection{Key Design Findings}

Over 19 iterations, $\Sigma$ produced the following engineering conclusions:

\begin{enumerate}[label=(\arabic*)]
    \item KNSB is viable for a 1\,km vehicle, with a total impulse requirement of approximately 15\,kN$\cdot$s.
    \item Aluminum 6061-T6 tube construction is cost-optimal for the 1\,km class (vs. carbon fiber, which offers marginal mass savings at 3$\times$ cost for this altitude).
    \item Parachute recovery (dual-deployment: drogue at apogee, main at 200\,m) is the recommended recovery strategy.
    \item A 4-fin trapezoidal configuration provides adequate static stability (stability margin $> 1.5$\,cal) across the Mach 0--0.6 flight envelope.
\end{enumerate}

\section{Evaluation}
% ======================================================================

\subsection{Error Detection}

Across 19 design iterations, $\Sigma$'s cross-review mechanism detected 34 qualitative issues (average 1.8 per RIGOROUS round). Of these, 11 (32\%) involved quantitative errors that would have propagated through a sequential agent pipeline without detection. The devil's advocate mechanism contributed 4 unique findings that no other agent identified independently.

\subsection{Convergence Behavior}

RIGOROUS-tier tasks converged in 2--3 rounds (mean 2.4, median 3) within the 4-round limit. One task hit the oscillation detector (agents alternating between aluminum and carbon fiber for the body tube material), resolved by the Director breaking the tie based on cost analysis.

\subsection{Cost Efficiency}

The complexity-adaptive system reduced costs by approximately 6$\times$ for LITE-tier tasks compared to running the full RIGOROUS protocol. A typical RIGOROUS rocket design round consumes \textasciitilde12,000 tokens (\textasciitilde\textyen 0.04 at DeepSeek V4 Pro pricing), making the economic barrier to iterative engineering design negligible.

\section{Related Work}
% ======================================================================

\textbf{Multi-agent frameworks.} CrewAI~\cite{crewai} popularized role-based agent orchestration but lacks structured review mechanisms. AutoGen~\cite{autogen} provides flexible conversation patterns but does not enforce information isolation during analysis. LangGraph offers graph-based control flow but requires manual definition of review edges.

\textbf{Multi-agent debate.} Du et al.~\cite{du2023debate} showed that multi-agent debate improves reasoning accuracy, but their approach uses homogeneous agents debating the same prompt rather than domain-specialized agents with complementary expertise.

\textbf{AI for engineering.} Recent work applies LLMs to engineering tasks~\cite{llm4engineering}, but typically in single-agent or simple sequential settings. $\Sigma$ contributes a protocol layer that adds adversarial quality assurance without requiring domain-specific training.

\textbf{Consensus mechanisms.} Distributed consensus algorithms (Paxos, Raft) address agreement on discrete values in distributed systems. $\Sigma$'s consensus estimation adapts this idea to continuous engineering estimates with calibrated uncertainty.

\section{Limitations and Future Work}
% ======================================================================

\begin{enumerate}[label=(\arabic*)]
    \item \textbf{LLM dependency.} $\Sigma$ inherits the reasoning limitations of its underlying LLM. While the protocol catches many errors, it cannot compensate for fundamental failures in physical reasoning. Future work should explore hybrid neuro-symbolic approaches where physics simulators serve as ground-truth validators.

    \item \textbf{Tool reliability.} Several RocketFactory tools (OpenRocket 23.09 GUI-only, FreeCAD on Windows) returned simulated/fallback data. The consensus estimation mechanism partially compensates, but ground-truth tool data would improve convergence quality.

    \item \textbf{Single-domain validation.} The current evaluation is limited to rocket engineering. Cross-domain validation (e.g., civil engineering, drug discovery) is needed to establish generality.

    \item \textbf{Human-in-the-loop.} $\Sigma$ currently places the human (Founder) as an external decision authority. Tighter integration---where the human participates as a first-class agent in the AERC cycle---could improve both safety and output quality for high-stakes applications.

    \item \textbf{Scaling.} The 8-agent configuration has not been tested at larger scales (20+ agents). Cross-review overhead grows as $O(n^2)$ in the number of agents, suggesting a need for hierarchical review structures in larger deployments.
\end{enumerate}

% ======================================================================
\section{Conclusion}
% ======================================================================

We presented Sigma ($\Sigma$), an AERC protocol-based multi-agent framework that introduces structured independent analysis, mandatory cross-review, and complexity-adaptive tiering to address the core failure modes of current multi-agent LLM systems: error propagation, groupthink convergence, and cost-rigidity. Through the RocketFactory case study---a real-world attempt to design a recoverable 1\,km rocket entirely via AI agents---we demonstrated that the protocol catches physically significant errors (a 16\% propellant performance overestimate) that escape single-agent and sequential multi-agent analysis. Sigma is open-source (AGPLv3), production-tested (985 tests), and available as a pip-installable Python package. We invite the community to apply, extend, and challenge the AERC protocol across engineering domains.
